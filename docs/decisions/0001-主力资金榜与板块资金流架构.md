# ADR-0001: 主力资金榜 + 板块资金流架构

**日期**: 2026-06-23
**状态**: ✅ Accepted
**作者**: 张胜东 + Hermes
**关联**: [README.md](../../README.md) / [collectors/community_enhanced.py](../../market_service/collectors/community_enhanced.py)

---

## Context

Financial-data-service (8084) 已稳定为 6 类核心数据采集 + 调度服务:
- 基金净值 / 估算净值 (`nav` / `estimate_nav`)
- 股票实时行情 + 日K / 分钟K (`price` / `daily` / `minute`)
- 大盘指数 (`index`)
- 北向资金 (`hsgt`)
- 财经新闻聚合 (`news_aggregator`)
- 社区数据 (`community_ / stock_news / xueqiu_hot`)

**缺失能力**: 主力资金流向。
- 张胜东的周报、操盘复盘需要"主力资金净流入榜"做资金面分析
- 盘前资讯、短期走势需要"板块资金流"判断资金面方向
- 现有 akshare 的 `stock_individual_fund_flow_rank` 接口稳定性差, 且无板块数据
- 东方财富 `push2.eastmoney.com` 的资金流接口(主力/板块)在浏览器/客户端可用, 但**对服务器数据中心 IP 主动断连**(反爬)

**问题**: 如何在 8084 体系下, 集成主力资金榜 + 板块资金流, 同时规避 EM 封禁, 并为 cron / 报告 / 复盘提供持久化数据源?

## Decision

**新增 1 张表 + 2 个 data_type + 3 个订阅**, 复用 `community_enhanced` collector (同源东方财富 API), 借 `push2delay.eastmoney.com` 绕过 IP 封禁。

### 三条核心决策

1. **复用现有 collector, 不新建** — `_fetch_main_fund_rank` / `_fetch_sector_fund_flow` 加在 `community_enhanced.py`, 因为它已有东方财富 UA + Referer + Session 复用 + 退避重试基础设施。

2. **持久化进新表 `fund_flow_data`** — 16 列, 主力资金榜 (`flow_type='main_fund_rank'`) 和板块资金流 (`flow_type='sector_fund_flow'`) 共表, 通过 `sector_kind` (`industry` / `concept`) 区分板块类型。`UNIQUE(source, flow_type, rank_data, name)` 实现按日去重。

3. **MCP 工具 + 8084 持久化双轨** — 盘前 9:00 cron 用 mcp_eastmoney 实时拉(8084 9:00 时还没数据, trading_hours_only 限制); 9:30 / 14:30 / 周六 20:00 的 8084 scheduler 自动入库, 供周报 / 短期走势 / 周复盘读取历史。

### 数据源选择

| 候选 | 数据完整度 | 服务器友好度 | 选不选 |
|------|----------|------------|--------|
| `push2.eastmoney.com` (主力资金榜) | ✅ 全 | ❌ 数据中心 IP 主动断连 | ❌ |
| `push2delay.eastmoney.com` (mcp-eastmoney 同源) | ✅ 全 | ✅ 延迟接口, 服务器可用 | ✅ |
| akshare `stock_individual_fund_flow_rank` | ⚠️ 偶尔空 | ⚠️ 偶发失败 | ❌ |
| 新浪 / 腾讯 / 通达信 | ❌ 无板块资金流 | ✅ | ❌ |

**结论**: 选 `push2delay.eastmoney.com` (跟 mcp-eastmoney 完全同源, 字段含义一致, 详见 ADR 链接)

### 表结构 (schema.sql)

```sql
CREATE TABLE fund_flow_data (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source TEXT NOT NULL DEFAULT 'eastmoney_em',
    flow_type TEXT NOT NULL,                    -- main_fund_rank / sector_fund_flow
    sector_kind TEXT,                           -- industry / concept (板块资金流填)
    rank_no INTEGER,                            -- 排名
    symbol TEXT,                                -- 个股代码 (主力资金流填)
    name TEXT NOT NULL,                         -- 名称 (个股/板块)
    price REAL,                                 -- 现价
    change_pct REAL,                            -- 涨跌幅 (%)
    main_net_inflow REAL,                       -- 主力净流入 (元)
    main_net_pct REAL,                          -- 主力净流入占比 (%)
    super_large_net REAL, large_net REAL,
    medium_net REAL,    small_net REAL,
    leading_stock TEXT,                         -- 领涨股 (板块资金流填)
    leading_change_pct REAL,
    rank_data TEXT NOT NULL,                    -- 排序日期 YYYY-MM-DD
    collected_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
    raw_data TEXT,
    UNIQUE(source, flow_type, rank_data, name)
);
CREATE INDEX idx_fund_flow_lookup ON fund_flow_data(flow_type, rank_data DESC, rank_no);
CREATE INDEX idx_fund_flow_cleanup ON fund_flow_data(collected_at);
```

### 字段映射 (东方财富 → 数据库)

| DB 字段 | EM 字段 | 转换 |
|---------|---------|------|
| `price` | `f2` | 浮点 (元) |
| `change_pct` | `f3` | **÷ 100** (存储为小数) |
| `main_net_inflow` | `f62` | 浮点 (元) |
| `main_net_pct` | `f184` | **÷ 100** (存储为小数) |
| `super/large/medium/small_net` | `f66/f70/f76/f78` | 浮点 (元) |
| `leading_stock` (板块) | `f128` | 字符串 |
| `leading_change_pct` (板块) | `f136` | **÷ 100** |

### 调度策略

| 订阅 | symbol | data_types | frequency | trading_hours_only | 何时读 |
|------|--------|-----------|-----------|-------------------|--------|
| 主力资金榜 | `__main_fund__` | `main_fund_rank` | 60min | **0** (全天) | 周报 / 短期 / 周复盘 |
| 行业板块资金流 | `__sector_industry__` | `sector_fund_flow` | 60min | **0** | 周报 / 短期 / 周复盘 |
| 概念板块资金流 | `__sector_concept__` | `sector_fund_flow` | 60min | **0** | 周报 / 短期 / 周复盘 |

**关键决策**: `trading_hours_only=0`。
- 主力资金榜 / 板块资金流在 9:00-9:30 之间 8084 scheduler 需要跑 (盘前资讯 cron 9:00 触发)
- akshare 9:00 跑没问题是因为它的 nav 接口交易时段外也跑
- 但我们用东方财富接口, 它"全天候有数据"(昨日 14:30 收盘后到今天 9:30 开盘前缓存的"最后一刻"数据)

## Consequences

### ✅ 收益

- **数据全**: 主力资金 + 行业板块 + 概念板块, 三个订阅
- **稳定**: `_get_with_backoff` 指数退避, 实测 push2delay 不限频
- **持久化**: 8084 SQLite, 周报 / 短期 / 操盘复盘都能 SELECT 历史
- **可扩展**: 后续要加"超大单/大单/中单/小单净额"分析 (字段已存) 不用再改 schema
- **双轨**: MCP 实时拉 (盘前 9:00) + 8084 持久化 (9:30 后) — 任何时候都有数据

### ⚠️ 风险

- **push2delay.eastmoney.com 限频**: 实测 60s 内多次并发会触发 (curl `Empty reply from server`)。`_get_with_backoff` 3 次重试, 不够的话需要降速
- **trading_hours_only=0 的副作用**: scheduler 在 22:00 / 02:00 等深夜也跑, 增加 EM 调用次数, 长期可能有 IP 风险。**缓解**: 60min 频率 + 退避重试, 实际每日 EM 调用次数 < 80
- **数据 schema 与 mcp-eastmoney 耦合**: 如果 mcp-eastmoney 改字段含义, 我们要跟着改。**缓解**: 字段转换逻辑只在 `_fetch_*` 方法里, 改一处即可

### 🔧 Pitfalls (实战踩坑)

1. **字段错位**: 第一次写代码误把 f2 (价格) 除以 100, 导致大族激光现价显示 1.42 元 (实际 145.11 元)。**根因**: 主力资金榜接口 f2 已是元单位, **不需除 100** (板块的 f3 才要除 100)。**修复**: 注释明确"f2=价格(元), f3=涨跌幅×100(%)"

2. **Circular reference**: collect() 里 `it["raw"] = it` 自引用, `json.dumps` 时崩溃, 数据**入库失败但 collection_log 标 success**。**修复**: raw 用 `{k:v for k,v in it.items() if k!="raw"}` 复制, 排除自引用

3. **trading_hours_only=1 的盲区**: 盘前 9:00 时 scheduler 还没跑 (9:30 才开始), 但盘前资讯 cron 9:00 需要数据。**修复**: 改 `trading_hours_only=0` + 9:00 走 MCP 实时拉 (双轨)

4. **EM IP 封禁 (push2)**: 第一次用 `https://push2.eastmoney.com` 测试, curl Empty reply, 确认是 IP 封禁。**修复**: 改 `push2delay.eastmoney.com` (mcp-eastmoney 用这域名, 同一接口)
